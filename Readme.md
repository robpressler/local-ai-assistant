# Local AI Assistant — Local-First Multi-Node AI Ecosystem

A local-first AI assistant system designed to help people build, understand, own, and shape their own AI tools safely, privately, honestly, and under human control.

---

## What This Is

This project is a locally hosted AI assistant ecosystem built around personal hardware, persistent memory, clear roles, and controlled growth toward more capable autonomous behavior.

It began as a single local assistant and is evolving into a multi-node system where separate AI nodes can communicate, check health, compare answers, log recommendations, and operate under shared rules.

The goal is not to replace human judgment. The goal is to give people more ownership over how AI works for them.

---

## Why Local-First Matters

Many AI systems make important trust decisions difficult for users to see or control.

This project takes a different approach:

- You should know where your data goes.
- You should understand when local AI is being used.
- You should know when cloud/API assistance is involved.
- You should control the tradeoff between privacy, performance, and convenience.

Local-first does not mean anti-cloud. It means the user gets to choose.

---

## Trust Modes

Different people need different trust levels.

### Local-Only Mode

Private data stays on hardware you control.

This is best for:

- personal notes
- sensitive files
- private projects
- local memory
- offline experimentation
- situations where privacy matters more than raw performance

### Hybrid Mode

External AI/API help can be used when the user chooses it and understands the tradeoff.

This is useful when:

- better reasoning is needed
- research or current information is needed
- local hardware is not powerful enough
- speed or quality matters more than keeping everything local

The important rule is disclosure and control: the user should know when hybrid help is being used.

## Current System

The project is organized around four named roles:

| System | Role |
|---|---|
| **RJ** | Primary user-facing assistant and control node |
| **Unknown** | Challenger and experimental node |
| **Continuum** | Memory, archive, and consistency-checking node |
| **Fred** | Gateway layer for phone access, routing, and future public beta use |

RJ, Unknown, and Continuum are the internal local systems.

Fred is the planned gateway between people and the internal AI network.

---

## Current Capabilities

Current and active-development capabilities include:

- local assistant runtime
- persistent conversation memory
- structured fact handling
- proposal logging
- file ingestion
- Docker-based deployment
- multi-node communication
- node health checks
- upgrade recommendation logging
- role-based system design
- standards-based safety and permission direction

The project is intentionally being built step by step rather than jumping directly into full autonomy.

---

## Roadmap

Planned development direction:

- improve local memory and state boundaries
- strengthen node-to-node communication
- expand health checks and failure handling
- improve upgrade recommendation workflows
- build Fred as the private mobile gateway
- add Fred query routing across nodes
- add answer comparison and synthesis
- create a public beta version of Fred
- separate private local use from public beta use
- gradually increase system autonomy only after safety, logging, and permission checks are stable

---

## What This Is Not

This project is not:

- a claim of completed self-awareness
- a claim of full autonomy
- a data brokerage project
- an anti-platform manifesto
- a replacement for human judgment
- a public release of private memory or personal logs
- a system that should perform external actions without permission

The project is experimental, but the goal is controlled experimentation.

---

## First Testing Direction

The next major public-facing step is Fred.

Fred is intended to become a simple way for others to try the system through a controlled interface. The first version will focus on beta-level testing, feedback, routing, and safe user interaction.

Public testing will be separated from private local memory and internal system data.

Fred is currently in development.

---

## Safety and Privacy Principles

Core principles:

- The user stays in control.
- Private data should stay local unless the user chooses otherwise.
- Hybrid/cloud assistance should be disclosed.
- External actions require permission.
- Memory should be intentional, not accidental.
- Disagreement between AI systems should be visible when it matters.
- The system should not pretend to know what it does not know.
- The system should grow gradually and safely.

The goal is AI that works with people, not around them.
